## 0. 一行コンセプト
実在の地形・地理・歴史を舞台に、まっさらな草原から「物流・交流」で文明を再興し、時代を越えて街・道・鉄道が富んでいくのを**育て・眺める**シミュレーション。進歩の速さを史実と競う。

## 1. 体験の芯(なぜ面白いか)
- SimCity 的な**育成・鑑賞** × 能動的**ビルダー**(自分で敷設・改良する。方針(ii))。
- 舞台が**実在の世界**(自分の街／日本／世界)。位置情報スタートで故郷を育てられる。
- 遊びながら**地理・歴史を学ぶ**(特産・名所・偉人・年代)。
- 育てた街が現実の地図とほぼ一致する**「答え合わせ」**のクライマックス。

## 2. 中心思想:繁栄=進歩のものさし
- 人口や財は時代で変わる相対値。**繁栄=進歩そのものを測る不変の物差し**とする。
- **物流・交流が進歩を推進する**。これが全メカニクスの背骨。

## 3. tier = 循環(交流)のスケール × 高度さ
tier(時代段階)は「到達した循環の**規模**と**高度さ**」で定義する2軸。
- **規模(到達半径)**:集落 → 郡/国 → 地方 → 全国 → 国際 → グローバル
- **高度さ**:物々交換 → 貨幣 → 市場 → 金融/信用 → 情報/デジタル

### 史実ラダー(訂正版)
| 段階 | 目安年代 | 循環スケール |
|---|---|---|
| 縄文 | 先史 | 集落＋遠距離の希少財(黒曜石・ヒスイ) |
| 弥生〜古墳 | BCE300〜 | クニ形成、大陸と初接触(希少) |
| 奈良・平安(律令) | 〜700〜 | **放射状の全国行政統合**(税が都へ)。横の市場流通は薄い |
| 鎌倉・室町・戦国 | 1200〜 | 地方経済圏の成熟＋**対外交易は活発**(日明・南蛮) |
| 江戸 | 1600〜 | **全国市場の統合**(天下の台所・五街道・海運)。対外は"四つの口"に限定(海禁) |
| 明治・大正 | 1860〜 | **開国=対外交流の本格化**＋鉄道・港で国内近代化 |
| 昭和 | 1920〜 | 貿易立国・高度成長(対外の大規模化) |
| 平成 | 1990〜 | グローバル化(金融・情報・SCの地球規模化) |
| 令和〜 | 2019〜 | 次の進歩(デジタル/宇宙/脱物質) |

**史実の要点訂正**:①対外交流は「昭和発」ではない——弥生に萌芽、室町戦国で活発、**明治で本格化**。②律令(奈良平安)は"放射状の全国統合"でスケールは大きいが横の市場は未発達。③江戸は対外を意図的に絞り(海禁)、国内統合を極めた段階。
→ 史実は単調な一直線ではないので、ゲームは**理想化した「循環拡大ラダー」を再走する if 世界**とし、現実マップとの答え合わせで史実のニュアンスを lore として見せる。

## 4. 発展の単位 = 地域(アーキタイプ)
- アーキタイプ(山＝信仰・木材・鉱／海＝漁・港・塩／川＝米・舟運／平野＝商業…)は**地形データから自動判定**(海岸距離・比高・河川/低地・用途密度)。
- **特産品**=そのアーキタイプ/土地に応じた産物(内部カテゴリの"皮")。
- **名所**=特別ノード(観光・信仰・解禁条件)。

## 5. 経済モデル(何を数えるか)
- **主計器 = 繁栄**(進歩の物差し。前面に出す指標)。
- **交易通貨 = 少数カテゴリの財**(例:食料/原料/製品/文化・信仰)＋**人(労働力)**。**フロー中心**(ルートを流れる量・生産消費レート)。在庫は小さなバッファのみ=倉庫管理はさせない。
- **収容力(carrying capacity)**:人口は地元資源で頭打ち。**財を輸入すると天井が上がる**=「孤立地域の限界 → 交流で飛躍」を機構化。
- **お金は任意**:交易黒字が建設予算として湧く形にすれば、コスト感を出しつつ財布管理は不要。
- 実在の特産名(茶・生糸・魚…)はカテゴリの皮 → 教育的かつ軽量。

### 5.1 財カテゴリの確定
**食料/原料/製品/文化・信仰の4カテゴリ＋人(労働力)を別枠**、で確定。
- **食料**:生存・人口の直接的な源泉。
- **原料**:生産のインプット(木材・鉱石・繊維など。特産品の"皮"の多くはここ)。
- **製品**:原料を加工したもの(消費満足・tier進行に効く)。
- **文化・信仰**:名所・偉人・儀礼など非物質。高度化型ブレークスルーの土壌にもなる。
- **人(労働力)**:別枠。生産の担い手であり、同時に食料の消費者。

### 5.2 収容力式(決定)
`収容力 = 地域ベース(アーキタイプ・地形由来) × (1 + 輸入ボーナス)`
- 地域ベースはその土地固有の食料生産力(平野は高い、山は低いが他資源で補う)。内発的ブレークスルー(治水・品種改良、§11.1の「収容力アップ」型)はここを直接押し上げる。
- 輸入ボーナスは`k × log(1 + Σ輸入フロー)`の**対数逓減関数**。単純比例だと天井が青天井になるため逓減させ、「孤立の限界は超えるが輸入だけで無限にはスケールしない」緊張感を保つ。
- **多様なカテゴリからの輸入ほど係数を優遇**(1種類を大量輸入するより、複数カテゴリを組み合わせる方が効率がいい)。

### 5.3 繁栄式(決定):主計器は「時代を跨いで比較できる物差し」
`繁栄(地域) = tier到達度(主要項) + 収容力に対する充足率(人口/収容力の比) + 交流多様性(繋がっているカテゴリ・相手の豊富さ)`
- **tier到達度**が主要な駆動源(時代が進むほど重みが増す)。
- **充足率**を入れることで、「縄文の村が身の丈に合って繁栄している」と「江戸の都市が身の丈に合って繁栄している」を**絶対人口ではなく同じ物差しで比較可能**にする。これが「人口や財は時代で変わる相対値、繁栄は不変の物差し」の実体化。
- **交流多様性**は孤立した肥大都市よりバランスの取れたハブを評価する。
- **役割分担**:この「繁栄」はUI上の主計器・鑑賞用の指標(地域ごと・全国集約)。史実と競う**公式スコア(累積リード、§9・§7.1)は、マップ最高tier地域(フロンティア)のtier到達だけを見る**、という切り分けのまま両方が並存する。
- **残課題**:k・各項の重み(w1/w2/w3)などの具体数値。

## 6. 進歩のエンジン:ブレークスルー(発明)
- **需要プル型**:欠乏・物流の詰まり=「社会の潜在的願望」→ 発明を呼ぶ。**コアの「詰まりを見つけて解く」ループがそのまま発明装置**になる。
- **確率だが steer できる**:願望(欠乏圧)＋蓄積パラメータ＋**異質地域との交流量**が高いほど発生確率が上がる。
- **交流が発明を加速**(idea diffusion。史実的にも正しい)。多様なネットワークほど速く進歩。
- 2種:**スケール解禁型**(次の半径を解禁。§7.2の交通tier表に新しい行を追加する)と**高度化型**(同じ半径の密度を上げる)。
- 発明は**ネットワークを伝って伝播**(強いリンクほど速く、孤立地域は遅れる)。

### 6.1 ブレークスルー候補リスト(例。網羅ではなくデータで拡張可能)
**スケール解禁型**(交通tierに新しい行を追加):稲作/農業定着、舟運(河川交通)、橋・渡船の改良(局所的な詰まり解消のミニ版)、運河、街道整備(五街道級)、帆船・港湾、鉄道、汽船、高速道路・自動車(内燃機関)、航空機・空港(旅客・高付加価値輸出、観光需要を解禁)、電信、コンテナ化、インターネット。

**高度化型**(同じ半径のまま密度アップ):文字・記録、郵便・飛脚制度、貨幣、市場制度(定期市→常設市場)、信用・為替、印刷、無線・ラジオ(情報のみのリンク。願望メーターの交流多様性ボーナスに直接効く)、冷蔵・コールドチェーン(生鮮品の実効交易半径を拡大)、銀行・金融機関、保険・証券、情報革命(コンピュータ)。

### 6.2 願望メーター:確率式の仕組み(決定)
各ブレークスルー候補ごとに「欠乏圧シグナル S」を定義する。
- スケール解禁型:S = (需要フロー − 現在の経路容量)÷需要フロー(詰まりの大きさ)。既存の「詰まりを見つけて解く」ループがそのままシグナル源になる。
- 高度化型:S = 制度的摩擦の大きさ(例:物々交換の不成立率)。

蓄積パラメータ(願望メーター、プレイヤーに見える0〜100%ゲージ)は `A += k × S × 交流多様性ボーナス` で毎ティック積み上がる。「交流多様性ボーナス」は繋がっている地域・財の種類の豊富さで決まる(「異質地域との交流が発明を加速する」の実体)。

ゲージが満タンになったら即発明ではなく、そこから先は毎ティック確率ロール(base_p×多様性倍率)に入る。プレイヤーの物流判断でゲージが早く満ちるほど平均的な発生タイミングも早まるが、満タン後も「いつ来るか分からない」余地が残る=「確率だがsteerできる」の実体。

**ネットワーク伝播**:既に別地域で発明済みのブレークスルーは、繋がっている地域の蓄積速度に「模倣ボーナス」を掛ける(近くに実例があると自地域の願望メーターが速く満ちる)。ゼロから発明する(フロンティア)より、既存のものを取り入れる方が速いという非対称性がこれで表現できる。

**残課題**:k・base_p・多様性倍率など具体的な数値。

## 7. tier の伝播:2つの regime
接続相手の tier で場合分けし、連鎖爆発を防ぐ。
- **フロンティア前進(世界の最高 tier を押し上げる)**:同等・異質の地域が持続的な循環で結ばれ、**ブレークスルー条件を満たした時に集団で N→N+1**。難しく・集団的で・発明と時計に律速される。("歴史を進める"のはここ。以前の「≧ゲート」はこの話。)
- **拡散・底上げ(既存の進歩を周縁へ広げる)**:自分より高い tier M の地域に繋ぐと、**時間をかけて徐々に、上限 M−1 まで**引き上げ。M(同値)にしないことで接続時の一斉+1連鎖を防止。**周縁は中心より常に1段遅れる**(史実的にも自然、地図がグラデーションになり鑑賞的にも良い)。
- 規則:**相手 ≦ 自分 → フロンティア規則／相手 > 自分 → 拡散規則**。

### 7.1 地域tierの内訳:規模(所属)と高度さ(成熟)を分離(決定)
tierの2軸(規模×高度さ)は、**更新の速さが違うので別々に扱う**。
- **規模(ネットワーク所属)**:物理的な接続で決まる。ただし**単なる到達可能性ではなく、一定以上のフローが継続して流れて初めて所属と判定**(下記7.2の経路容量が前提)。所属先(接続相手)の規模が上がればこちらの上限も追いかけて動く**動く上限**(固定しない)。
- **高度さ(制度の成熟)**:規模所属が決まった後、既存の拡散規則(M−1へ徐々に)で追いつく。
- **複数接続がある場合**:一定以上のフローが流れている接続の中で**最大のM−1を採用**(弱い接続は無視、強い接続だけが底上げに効く)。
- **視覚的な即時フィードバック**:経路をアップグレードした瞬間、容量の伸び幅に比例して交易フローが即座に急増する演出(いわゆる「宿場町ブーム」)。これは規模所属の正式判定(閾値超え)より先に見える。正式な規模所属・tier反映は閾値を超えてから。
- **スコアとの接続**:プレイヤー文明全体のtier(史実基準線と比較する対象)は**マップ上の最高tier地域(フロンティア)を採用**。周縁地域の追いつき具合は主スコアではなく副次的な鑑賞指標として見せる。

### 7.2 経路種別 = 交通tier(規模上限×フロー容量×移動速度の3点セット、決定)
経路(道・鉄道・航路など)の種別ごとに、到達しうる最大スケール・運べるフロー容量・移動速度の3つをまとめて規定する。§6のスケール解禁型ブレークスルー(稲作・帆船/港湾・蒸気/鉄道・汽船・電信・コンテナ・内燃機関・ネット)が、そのままこの表の新しい行(=新しい上限の交通tier)をアンロックする。

| 経路種別 | 最大到達スケール | フロー容量 | 移動速度 |
|---|---|---|---|
| 獣道・畦道 | 集落内どまり | 小 | 遅い |
| 街道(未舗装) | 郡・地方まで | 中 | やや遅い |
| 五街道級・舗装街道 | 全国まで | 中〜大 | 中 |
| 鉄道 | 全国〜国際の起点 | 大 | 速い |
| 港湾+帆船/汽船 | 国際 | 大(海運は元来大容量) | 中〜速い |
| 電信・コンテナ・ネット | グローバル | 情報/物流とも極大 | 実質瞬時(情報)/大幅短縮(物流) |

低い経路種別のままだと、どれだけフローを流そうとしても容量の天井に阻まれて上位スケールの所属閾値を物理的に超えられない(獣道のままでは永遠に全国スケールへ正式昇格しない)。ちゃんとした経路への投資が規模拡大の必須条件になる、という設計。
**残課題**:所属判定の具体的なフロー閾値、経路種別ごとの容量・速度の具体数値。

## 8. 移動速度 = 時間距離の短縮(time-space compression)
- 移動速度は**経路の交通 tier に比例**。序盤に気が遠くなるほど遠い場所が、街道→鉄道を通すと一気に近くなる。
- **(a) プレイヤー分身の3D移動**(世界の大きさを体感)と **(b) 物流フローの辺速度**(スループット)の**両方**に効かせる(両方とも経路の交通 tier で一貫)。
- 速さと到達半径が同時に伸び、**遊びの舞台が進歩とともに広がる**(集落→全国→グローバル)。
- **ファストトラベルは築いたネットワークで解禁**(既存のマップジャンプを「十分なインフラで繋がった所にだけ跳べる」へ進化)。
- 注意:縮みすぎ対策として、速度が伸びるぶん engage 範囲も広げ、常に外側にフロンティアを残す。

## 9. 時計とスコア
- **時計はアプリを開いている間だけ進む**。閉じている間は完全停止。「寝ている間・離れている間に置いていかれる」という現実カレンダーに紐づく強迫観念を排除する。スコア・史実基準線との比較も**プレイ時間(セッション内経過時間)**を軸に行い、現実の日数とは無関係。
- **進み方はリアルタイム連続シミュレーション(SimCity型)が基本**。一時停止・倍速(1x/2x/3x)で速度をプレイヤーが制御する。ターン制の「意思決定→時間送り」構造は取らない。
- **開いたまま放置(離席)している間は自動でアイドル速度(低速)へ落ちる**。閉じれば完全停止/開いて操作していれば通常〜倍速/開いたまま離席なら緩やかに進む、という3段構え。閾値・アイドル速度の具体値は未確定。
- **節目(ブレークスルー成立・tier到達など)では自動で一時停止し、意思決定ポイントを提示**(信長の野望的な重み付け)。連続シミュレーションの中に要所だけ区切りを挟む形で、純粋なターン制にはしない。
- **放置成長モード(オプション・デフォルトOFF)**:閉じている間も低速で発展させたい人向けに、任意ONの「留守中おまけ」機能を用意してよい。設計方針:①上限キャップを設ける(例:最大12〜24時間相当まで。それ以上は貯まらない=長く放置しても損も得もしすぎない)。②仕組みは常駐シミュレーションではなく、**再訪時に経過実時間からキャップ付きでまとめて計算**する方式(オフラインゲームの「お帰りなさい」方式)。既存サーバーに常駐プロセスを足す必要はない。③**この放置分の進行は史実基準線とのレース(累積リード)には算入しない**——あくまで鑑賞用のおまけであり、"サボらず前倒しした"という評価軸とは別勘定にして、抜け道にしない。
- **基準線 = 史実の進歩曲線**(日本、または人類共通)。「そのプレイ時間(圧縮年表上の年)に史実はどの tier だったか」。放置成長モードで進んだ分はこのプレイ時間に含めない。
- **スコア = 累積リード(前倒し年数)**=あなたの到達曲線と史実曲線の差の面積。加えてマイルストーン(「蒸気機関に80年早く到達」)と**常時リード/ラグ計**。基本方針として確定。ただし**地域集約の仕方(§13 論点3)を先に決めてから、単位・積分方法の詳細設計に入る**(複数地域のtierをどう一本の「プレイヤー曲線」に集約するかが前提になるため)。
- 時間は**非線形圧縮**(古代は早送り、近代以降は細粒)。プレイヤーの物流判断が加速装置、時計(セッション内自走)がプレッシャー——ただし閉じれば止まるので長期的な不安にはならない。
- **スノーボール対策**:史実基準線も後半で加速／拡散に時間がかかる／平均を追いつかせる。
- **失敗**:基本はソフト(スコア低下)＋循環の破局で**局所的 tier 後退=暗黒時代**。
- **終わり**:現在(present)に追いつく=**答え合わせのクライマックス**→以降は未来 tier エンドレス。
- **二モード**:史実レース(時計自走・採点)／サンドボックス(時計は手動・失敗なし)。中身は共通。

## 10. どこから始めても「地域に合った発展」を(全国・世界対応)
- **原則:メカニクスは普遍、コンテンツはデータ駆動。** 仕組みは全国共通、地域固有(特産・名所・偉人・名前)はデータから解決しハードコードしない。
- 地域の単位=市区町村(行政界)。アーキタイプ=地形から自動。特産・名所・偉人=Wikidata/Wikipedia、または手作り。
- **データ欠損時はアーキタイプ由来の汎用でフォールバック**(海→漁、山→木材/信仰)。破綻させない。
- **開始地点はプレイヤーが選べる**(位置情報検出は初期提案の一つに過ぎず、地図検索・候補都市リストから自由に選べる。詳細は§15)。世界対応は同じデータ駆動を海外へ(コンテンツは薄くなる。まず日本で厚く。ただしゲームとして遊べる強度は場所によらず均一を担保、§15.2)。

## 11. 歴史人物 = 時代連動の偉人効果
- 地域の段階が人物の史実の時代に達すると「覚醒」し、史実に沿った効果を発揮。
- 効果が史実の功績を教える → 遊ぶだけで通史が入る。

### 11.1 効果の型(決定):新メカニクスは作らず、少数の型+データで拡張
偉人効果の中には「物流・交流が進歩を推進する」という中心思想の**例外(内発的な進歩)**にあたるものがある(例:二宮尊徳の荒地再生は外部交易を介さない)。これを個別に新しいメカニクスとして作り込むと複雑になるため、**少数の汎用「効果の型」を用意し、偉人は型を参照するデータの1行として増やす**方針にする(§10の「メカニクスは普遍、コンテンツはデータ駆動」の原則そのもの)。新しい偉人を追加するたびにコードを書く必要はなく、Wikidataや手作りでデータ行を足すだけでよい。

効果の型(初期セット。今後データで拡張可):
- **収容力アップ**:地元の人口天井を直接引き上げる(例:二宮尊徳=荒地再生)。
- **経路容量/交通tier先取り**:インフラを史実より早く整備したことにする(例:太田道灌=街道・城下整備、武田信玄=治水)。
- **高度さブースト**:制度・経済の成熟度を先取りする(例:伊藤博文=明治近代化、渋沢栄一=実業/金融)。
- **地図拡張**:未踏エリアを解禁する(例:伊能忠敬=測量)。

| 偉人 | 地域 | 覚醒条件(到達era) | 効果の型 | 効果の中身 |
|---|---|---|---|---|
| 太田道灌 | 伊勢原 | 室町 | 経路容量/交通tier先取り | 街道・城下の交通tierを1段先取り |
| 二宮尊徳 | 小田原 | 江戸 | 収容力アップ | 荒地再生で人口天井+X% |
| 伊藤博文 | 大磯 | 明治 | 高度さブースト | 近代化制度を先取り |
| 武田信玄 | 甲斐 | 戦国 | 経路容量/交通tier先取り | 治水で被害を抑え生産を安定化 |
| 渋沢栄一 | (実業/金融拠点) | 明治 | 高度さブースト | 金融・実業制度を先取り |
| 伊能忠敬 | (全国) | 江戸後期 | 地図拡張 | 未踏エリアの解禁 |

地方の偉人をこの表に追加していくことで、体系を複雑にせず歴史学習のカバー範囲を広げられる。**残課題**:効果の型ごとの倍率・持続時間などの具体数値、型が5つ目以降必要になるかの見極め。

## 13. UI設計(決定)
- **2レイヤーのビュー切り替え**:①**3Dビュー**(既存の伊勢原ファンタジー由来)=地上を歩いて建物・地形・時代ビジュアルを眺める没入・鑑賞用の層。②**戦略マップ**(新規・俯瞰)=地域をノード、道・鉄道・航路を線で表現する俯瞰図(信長の野望の国地図やTransport Fever/Anno 1800の交易路表示に近い)。ノードの色・大きさでtierと繁栄を、線の太さ・流れるアニメーションでフロー量を表現。ルート敷設・地域選択・詳細確認はここで行う。切り替えは**明示的なボタン**から始める(シームレスなズーム連動は実装コストが高いので後回し)。
- **フロー表示**:各ルートに矢印/粒子アニメーションを流し、太さ・速度で流量と交通tierを表現。財カテゴリ別に色分け(食料=緑、原料=茶、製品=青、文化・信仰=紫)。容量に対して需要過多な詰まり箇所は赤ハイライトで可視化(=願望メーターが育っている場所)。
- **リード/ラグ計**:画面端に常時表示する小ウィジェット(現在の累積リード年数+トレンド矢印)。クリックでマイルストーン一覧・tier進行の史実比較グラフに展開。
- **アーキタイプ/飛躍候補の提示**:地域選択でサイドパネルを開き、アーキタイプ情報と現在の願望メーター一覧(候補ブレークスルーごとの進捗バー、確率的な期待値表示)を見せる。
- **節目の意思決定ポイント**:自動一時停止時に信長の野望のイベント風モーダル(背景暗転、選択肢2〜3個)を表示。選ぶと即座に反映され連続シミュレーションが再開する。
- **実装メモ**:戦略マップは既存3Dエンジンと別の新規コンポーネント。`HANDOVER.md`のソース分離方針(段階的抽出)と合わせて、2D(SVG/Canvas)の軽量な戦略マップは3Dエンジン抽出より先にプロトタイプできる可能性がある(実装フェーズの話として書き留め)。

## 14. 3D探索そのものを楽しめる要素(決定)
実マップ×実地形3Dという既存アプリの強みを、文明育成レイヤーの差別化に直結させる。
- **時代ビジュアルモードの tier 連動**:既存の5モード(現実/明治/江戸/メルヘン/宇宙)を手動切り替えから、**各地域の tier に応じて自動的に景観が変わる**仕組みに格上げする。マップ上を歩くだけで、進んだ地域と遅れた地域の間の「時代のグラデーション」を物理的に体験できる。実地形3Dだからこそ成立する、最も差別化になりうる要素。
- **経済シミュレーションの"目撃"**:詰まっている橋・渡し場には荷を積んだNPCの荷車や行列が見える、繁栄している地域は賑わい(人出・建物密度)で表現される、というように、戦略マップ上の抽象的な数値(願望メーター・フロー詰まり・繁栄)を3D空間内の情景として翻訳して見せる。
- **スカウト(探索の実用化)**:戦略マップのデータだけに頼らず、実際に3Dで歩き回って地形・資源(木立・鉱脈・良い港の入江など)を見つけることが、どこに経路を敷くかの判断材料になる。探索そのものにゲームプレイ上の意味を持たせる。
- **保留(不採用ではなく今回は据え置き)**:名所・史跡を訪れる"発見"コレクション要素は良さそうだが今回は見送り、実距離を歩く体感(§8)は既存設計に内包済みのため独立項目にはしない。

## 15. 初期集落の生成規則・データ源(決定)

### 15.1 開始地点はプレイヤーが選べる
位置情報による自動検出は「初期提案の一つ」に留め、**地図検索や候補都市リストから自由に開始地点を選べる**ようにする。故郷を育てたい人はそれを選び、興味のある土地(観光地・歴史的地域など)で始めたい人はそれを選べる。

### 15.2 「どこで始めても同じようにプレーできる」の担保
**コンテンツの厚み(実在の特産名・名所名・偉人)には差があってよいが、ゲームとして遊べるかどうかには差があってはいけない**、と切り分ける。
- Wikidata検索で名所・特産・偉人がヒットしなかった場合でも、**アーキタイプ由来の汎用コンテンツを必ず生成**し、その地域が「空っぽ」にならないようにする(例:山なら「名もなき鎮守の森」、海なら「名もなき漁港」)。
- **偉人はボーナス要素であって必須要素にしない**:偉人がゼロの地域でも、循環・フロー・ブレークスルーという核となるループ(収容力・tier進行・スコア)は完全に成立する。偉人効果は「あればラッキー、歴史も学べる」という上乗せ。
- 結果として、**故郷を選べば厚みのある物語体験、無名の土地を選べば薄いが同じ強度のゲームプレイ**、という差だけが残る。メカニクスの均一性(§10)の延長。

### 15.3 初期集落の生成規則
- 全国どこでも同じ規則:市区町村(または大字)の代表点を「シード地点候補」とし、アーキタイプは地形から自動判定。
- ゲーム開始時に姿を見せているのは**開始地点周辺の少数のシードだけ**(相模スライスなら大山・相模川・平塚大磯・小田原・秦野の5点程度)。それ以外の全国のシードは「未発見」状態で潜在的に存在し、探索(§14スカウト)や交通網の伸長に応じて徐々に姿を現す——「まっさらな草原から」という開幕コンセプトと全国・世界対応を両立させる。
- シードの間隔は「歩けなくはないが序盤は骨が折れる」距離感を目安に調整(具体的な閾値は残課題)。

### 15.4 データ源(ハイブリッド方針)
- **相模スライス(最初の実装対象)は手作り**:精度と物語性を優先し、既に挙げたデータ(大山阿夫利神社、二宮尊徳など)を丁寧に作り込む。
- **全国・世界への拡張はWikidata自動抽出+アーキタイプ由来のフォールバック**:
  - **偉人**:SPARQLで「その市区町村に生まれた/関連した人物」を職業タグで絞り込み、**職業キーワードから§11.1の効果の型へ自動マッピング**する(治水・土木系→収容力アップ、築城・街道整備系→経路容量/交通tier先取り、実業・金融・行政系→高度さブースト、測量・地理学系→地図拡張)。
  - **特産**:Wikidataの産物データや地理的表示(GI)保護制度のような公的リストがあれば利用、無ければアーキタイプ由来の汎用名(海→魚、山→木材)にフォールバック。
  - **名所**:Wikidata/Wikipediaの観光地・史跡・神社仏閣タグを利用。
- **精度にグラデーションがあることを許容**:相模は厚く、全国は自動生成で粗く、海外はさらに薄く、という考え方をデータ源選定にそのまま適用する(ただし§15.2の通り、ゲームとして遊べる強度は均一)。
- **残課題**:シード間隔の具体的閾値、Wikidata SPARQLクエリの具体設計、職業→効果の型マッピングの詳細ルール。

## 16. アーキテクチャ:データフローと保存方式(決定)
ゲームがどう動くかを2層に分けて整理する。

### 16.1 コンテンツ生成(地域の中身。一回きり・キャッシュ)
- アーキタイプ(海/山/川/平野)は地形データ(海岸距離・標高・河川・用途密度)から判定。特産・名所・偉人はWikidataクエリで取得。ヒットしなければアーキタイプ由来のフォールバック(§15.4)。
- **そのシードが最初に発見・接続された時に一度だけ計算**され、以後は基本的に変わらない静的データとして扱う。
- 既存の`server/server.js`が持つOSM・標高のプロキシ+ディスクキャッシュの仕組みをそのままWikidataにも拡張する形で実現できる(新しいインフラは不要)。
- **注意**:Renderの`cache/`は再起動で揮発する。相模スライスのように厚く作り込む対象は、キャッシュに頼らず**事前生成してリポジトリにコミット**しておく(Wikidataは頻繁に変わらないため)。

### 16.2 シミュレーションループ(文明の進行。継続的)
- tier・繁栄・フロー・願望メーター・偉人の覚醒判定・累積リードは、**アプリを開いている間、ティックごとに継続して計算**される動的な状態。
- **クライアント側(ブラウザ内)で実行**するのが自然:①既存アプリは three.js の単一クライアントで、フレーム更新ループに同居させやすい。②「常駐サーバープロセスを持たない」方針(§9)と一致する。サーバーは引き続き地形・Wikidataのプロキシ+キャッシュ役に徹する。

### 16.3 セーブデータ(決定:ブラウザ内で完結)
- プレイヤーのセーブデータ(tier進行・発見済みシード・覚醒済み偉人・繁栄・時計・スコア)は**ブラウザの`localStorage`に保存**し、サーバー側データベースは持たない。端末をまたいだ継続はスコープ外とする(1台のブラウザで完結する前提)。これにより実装がシンプルに保たれ、「常駐サーバーを持たない」方針とも一致する。
- 将来的に端末をまたいだセーブが必要になった場合は、アカウント+サーバー側DB(SQLite/Postgres等)の追加が必要になる、という認識だけ残しておく。
