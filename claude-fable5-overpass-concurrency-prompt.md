# 相談プロンプト: Overpass APIの同時実行数設定は妥当か

以下をそのままFable 5に貼ってください。文脈: これまでNYでのクラッシュ調査(facadeCache参照カウント等)と、
移動中に道路・建物の生成が止まる問題(429/502/504のグローバル・クールダウン、gaveUp判定の再設計)を
別チャットで段階的に修正してきた。今回、そもそも論として「OSM取得のエラー自体がこんなに出るのは
正常なのか、ゲーム側の設定に問題があるのではないか」をユーザーに問われ、Overpass API公式ドキュメントを
調べたところ、コード側の設定と過去の実機検証コメントの間に矛盾を見つけた。この解釈と対策案が
妥当か、第三者の目で検証してほしい。

## 前提: このゲームのOSM取得方式

`js/legacy/part8.js`の`fetchOSMTileBatch`が、プレイヤーの周囲を1600m四方のタイルに分割し、
ブラウザから`https://overpass-api.de/api/interpreter`へ直接POSTでクエリを送っている(サーバー経由の
プロキシは介在しない、既に確認済み)。同時実行数は`OSM_TILE_CONCURRENCY = 3`(3並列)。

## 見つけた矛盾

`part8.js`冒頭、`OSM_TILE_CONCURRENCY`宣言の直前に、過去の実機検証履歴を記したコメントがある(要約):

> Overpass側の実際の同時実行枠は公式に2/IPと明言されている。2026年時点でもgall/lambertの2台が
> それぞれ独立してrate limit=2を課している。3のままだと3本目が恒常的に429になり…実際の上限に
> 合わせて2に戻す。
> 【2026-07-16】2→3。直接モードはミラー輪番(3ホスト)になったため、ホストあたりの同時実行は
> 従来以下のまま全体スループットを上げられる。
> 【2026-07-19・実験→撤回】private.coffeeメイン化に合わせて5へ上げたが、実機Renderログで
> private.coffee/kumi.systemsが常時タイムアウトし、実質ほぼ全リクエストがoverpass-api.de
> (公式に2並列/IPの制限)に集中していたことが判明。3のままの方が安全なので戻す。
> `const OSM_TILE_CONCURRENCY = 3;`

読み取れる経緯: 「2並列が実測での正しい上限」という認識自体は過去にあった。その後、
「3ホストへミラー分散するから3並列でも1ホストあたりは2以下に収まるはず」という理屈で3へ上げた。
ところがミラー(private.coffee/kumi.systems)は不調で使い物にならず、実質ほぼ全リクエストが
overpass-api.de(2並列/IPの制限)1本に集中していることが判明した。それにもかかわらず、
5からの撤回時に「3のままの方が安全」として3を維持し、本来の2には戻さなかった。
この「3のままの方が安全」は5との比較であって、2との比較ではない。

**現状の解釈**: ミラー分散が事実上機能していない今、「2スロットしかない単一ホストに3並列で
リクエストを送り続けている」状態になっており、これが移動中に429が多発する一因になっている
可能性が高い。

## Overpass API公式ドキュメントで確認した仕様(2026-07時点)

出典: https://dev.overpass-api.de/overpass-doc/en/preface/commons.html

- レート制限は「スロット」方式。1リクエストは実行時間+クールダウン時間ぶんスロットを占有する。
  クールダウン時間はサーバー負荷に比例して伸びる(低負荷時は実行時間の一部、高負荷時は数倍)。
- ユーザー(IPアドレス単位、IPv4はフルアドレス)に割り当てられる利用可能スロット数は
  `https://overpass-api.de/api/status` の `Rate limit:` 行に出る(固定の公開定数ではなく可変)。
- 「slippy map(地図をスクロールし続けて短いリクエストを連発するアプリ)」は名指しで想定利用例に
  挙げられており、スロットが埋まっているリクエストは最大15秒までサーバー側でキューされ、
  それでも空かなければ429で拒否される。ドキュメントは「429が来たら(まだその場所を見ている
  なら)15秒待って再送すべき」と明記している。
- 429はレート制限由来。504は**別の仕組み**(宣言したtimeout/maxsizeがサーバーの残りリソースに
  対して大きすぎる場合)由来で、レート制限とは無関係。

この記述から、「移動し続けるプレイヤーが時々429を受けること自体」はOverpass側が織り込み済みの
正常動作と読み取れる。一方で、恒常的・大量の429(実機ログでは数十秒〜数分単位で頻発)は、
単なる「公式が想定する通常のslippy map挙動」を超えて、**割り当てスロット数を超過した同時実行**が
原因になっている可能性がある。

## 追記: 開発者環境(Windows/Node)での実測結果(2026-07-20)

VM(サンドボックスLinux環境)が使えないため、開発者自身のPCでNode.js(v24)から
overpass-api.deへ直接検証した。TLS/VPN起因の接続エラーを除去した後の結果:

1. **単発・逐次(間隔は手動コマンド入力程度、数秒〜十数秒)を3回**:
   200 (1.51s) → 200 (2.73s) → **429 (11.61s)**。
   3回目は「並列」ではなく1本だけ送っているにもかかわらず429になり、しかも即座の拒否ではなく
   11秒待たされた末の429だった(ドキュメントの「最大15秒キューしてから429」の挙動と一致)。
2. **2並列(`Promise.all`で完全同時発射)**: 2本とも429(約8.3秒)。
3. **3並列**: 3本とも429(約8.6秒)。2並列と3並列で有意な差は見られなかった。
4. 3の直後に`/api/status`を確認したところ:
   ```
   Connected as: 1905960282
   Announced endpoint: lambert.openstreetmap.de/
   Rate limit: 2
   2 slots available now.
   ```
   「2 slots available」と空きがあるように見えたが、直前のPOSTとは`Connected as`のIDも
   `Announced endpoint`(lambert)も異なっており、**直前にPOSTを処理したのは別の物理サーバーだった
   可能性がある**(overpass-api.deは複数バックエンドへのDNSラウンドロビン構成と思われ、
   セッション冒頭の`/api/status`確認時は別のID・別のホスト名が返っていた)。

この結果から、「2並列に絞れば429が収まる」という単純な仮説は実測上支持されなかった
(2並列でも3並列でも同じく全滅した)。むしろ、クライアント側から制御できない
バックエンド振り分け・全体負荷の変動が支配的で、並列数の微調整よりも
「どのサーバーに当たっても429/504を静かに乗り切るクールダウン処理(実装済みの修正1)」の方が
本質的な対策ではないかという見立てに傾いている。

## 相談したいこと

1. 上記の「ミラー分散が機能していないなら2並列に戻すべき」という解釈は妥当か。見落としている
   反証(例えば公式スロット数が2から増えている可能性、直列化のオーバーヘッドの方が支配的な
   可能性等)はあるか。
2. `OSM_TILE_CONCURRENCY`を3→2に下げる対策は、429の**発生頻度そのもの**を下げる根本対策として
   優先度をどこに置くべきか。既に実装済みの対症療法(429/502/504のグローバル・クールダウン、
   gaveUp判定の再設計)との関係も含めて助言がほしい。
3. 2並列に下げた場合のトレードオフ(スループット低下でタイル取得がさらに遅くなる懸念)は
   どの程度深刻か。1タイルあたりのクエリ複雑度・バッチサイズ側で相殺できる余地はあるか。
4. 504対策として、バッチサイズ(現状: 通常3タイルまとめ、密集地や再試行時は1タイル)や
   クエリの[timeout:]/[maxsize:]宣言値を見直す余地はあるか。

コードの実装自体は別チャットで行う前提。まずは方針の妥当性だけ検証してほしい。
