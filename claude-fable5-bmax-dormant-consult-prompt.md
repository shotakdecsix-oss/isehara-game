# 相談プロンプト: bMax飽和時にdormantBuildingsが無制限に増え続ける問題

以下をそのままFable 5に貼ってください。文脈: 直前に別チャットで、Overpass APIの429/504対策
(OSM_TILE_CONCURRENCY 3→2、429時に/api/statusを読んで待ち時間を正確化、クエリの宣言timeoutを
保守的に短縮)を実装・デプロイした(あなたが以前書いたIMPL_PROMPT_20260721_OVERPASS_COSTの
1+3+2aに相当、2bと2cは実応答サイズ未確認・構造変更が大きいため保留)。この実機検証中に、
Overpassとは別の深刻な問題を示す数値が出た。

## 実機で観測した数値(2026-07-21、東京駅周辺と思われるstd設定)

デバッグオーバーレイの定期ログ:
```
[fetch] active 0 queue 65 cooldown(ms) 51695 streak 3 records 12000 / 12000 dormant 115180
```

`records 12000 / 12000` = 描画済み建物数がPERF.bMax(std設定の上限)に完全到達。
`dormant 115180` = dormantBuildings配列(未生成・退避中の建物の軽量記述)が11万5千件超。

Overpass側(cooldown/streak)は今回の対策通りの動作に見える(429/502/504が3連続でクールダウン中)。
一方dormantの数字は今回の対策と無関係に既存していた問題で、放置できない規模だと判断した。

## 関連コード(現状。line番号は`js/legacy/`基準)

- `part1.js:594` `PERF.std.bMax = 12000`(lite=6000, high=25000)
- `part1.js:778-838` `unloadFarBuildings(force)`: プレイヤーから一定距離を超えた実建物を
  解放し、軽量記述を`dormantBuildings`へpush。上限の95%を超えると(`_nearCap`、788行目)、
  消去距離を生成距離まで詰め(789行目)、さらに距離ヒストグラムで「近い順にbMaxの85%が
  収まる半径」を計算してそれより外を解放する(796-813行目)。
- `part1.js:850-867` `reactivateNearbyDormantBuildings()`: dormantBuildingsを低頻度
  (~1.5秒毎)で線形走査し、生成距離内に戻ってきたものをpendingBuildingsへ戻す。
  **855行目: `if (buildingRecords.length >= PERF.bMax) return;`** — 上限到達中は
  この関数が何もせず即returnする(スキャン自体をしない)。
- `part9.js:456-469` pendingBuildings処理ループ内: 実建物候補が生成距離内でも、
  `buildingRecords.length >= PERF.bMax`なら**pendingBuildingsから取り出した直後に
  そのままdormantBuildingsへpushしてcontinue**(465-468行目)。
- `part8.js:143` `if (seenOSMWays.has(el.id)) return;` — OSMのway単位で一度処理した
  IDは、同じタイルを再取得しても二度と処理されない(重複排除)。dormantBuildingsに
  積まれた建物の元データはこの時点で既に「処理済み」扱いになっている。

## 見立てている因果(検証してほしい)

std上限12000に対し、東京駅のような真に高密度なエリアでは「プレイヤー近傍だけで」
上限を超える実建物が存在しうる。この状態で:

1. `_nearCap`のヒストグラム縮小(789-813行目)は「近い順にbMaxの85%を残す」処理だが、
   その場に留まる限り、残りの15%相当は元々「そこそこ近い」建物であり、突出して
   遠い建物が別に存在しない限りunloadFarBuildingsは実質何も解放できない
   (=枠が空かない)。
2. 枠が空かない間、新しく届くOSMタイルの建物候補は469行目で問答無用にdormant行き。
3. reactivateNearbyDormantBuildingsは855行目のガードにより、上限到達中は**一切
   スキャンしない**ため、dormant→pendingへの復帰も完全に止まる。
4. 結果、プレイヤーがその高密度エリアに留まる(または同程度に高密度な別エリアを
   巡る)限り、dormantBuildingsは「一方通行の吸い込み口」になり、OSMタイルが
   届くたびに増え続ける。11万5千件はこのシナリオと整合する。

つまり「上限に達したら退避させ、離れたら戻す」設計自体は、上限が全体的に緩やかに
超過するケース(広いエリアを移動して合計が積み上がるケース)には対応できているが、
「ローカルに上限を超える密度」のケースでは吸い込むだけで一切吐き出さない構造的な
片道弁になっている可能性がある。

## 未確認・懸念点

- dormantBuildings配列自体に上限が無い。上記の一方通行シナリオだと理論上無制限に
  増え続ける(メモリ+reactivateNearbyDormantBuildingsのO(n)線形走査コストが
  上限到達中は発生しないので直接のフレーム落ちは今のところ無いはずだが、
  上限を抜けた瞬間に11万件のスキャン+大量のpendingBuildings流入が発生する懸念)。
- 単純に配列を切り詰める(古い/遠い順に間引く)対策は、`seenOSMWays`が該当wayを
  既に処理済み扱いにしているため、間引かれた建物はタイルを再取得しても二度と
  復元されない(=恒久的な欠落)可能性が高い。安全に間引くには`seenOSMWays`側の
  取り消しも同時に必要か、そもそも別のアプローチが要るかを判断してほしい。
- bMax自体を上げる対策は、project memoryにある「GPU crash & caps」の経緯
  (東京駅で70k道路メッシュ+建物無制限がクラッシュの直接原因だった)と衝突する
  可能性があるため、単純な引き上げは避けたい。

## 相談したいこと

1. 上記4の「一方通行の片道弁」という見立ては妥当か。見落としている経路
   (実は別のタイミングでdormantが減る箇所がある等)はあるか。
2. 最も安全な対策は何か。候補として: (a) 上限到達中でもreactivateNearbyDormant
   Buildingsを完全停止せず、evict(遠い実建物を解放)とrevive(近い建物を復帰)を
   セットで少量ずつ回す「入れ替え」を許可する、(b) dormantBuildings自体にハード
   上限を設け、閾値超過分は間引く(ただしseenOSMWaysとの整合が課題)、(c) 上限到達を
   検知したら該当エリアのOSMタイル取得自体を一時的に止める(そもそも吸い込み口を
   閉じる)、(d) その他。
3. 対策を入れる場合、既存のヒステリシス・GPUクラッシュ対策(距離キャップ・
   メッシュ数キャップ)との相互作用で新たな不具合を生まないための注意点は何か。

コードの実装自体は別チャットで行う前提。まずは原因の妥当性と対策の方針だけ検証してほしい。
