# 実装プロンプト: Overpass取得の総量管理(429根本対策)

前提: 別チャットでの分析・検証済み。並列数(OSM_TILE_CONCURRENCY)の微調整は本質ではない。
実測で「逐次1本でも3本目は429」を確認済み = 支配要因は並列数ではなく
**スロット時間(クエリ実行時間+クールダウン。クールダウンはクエリコストと負荷に比例)**。
このゲームのクエリは重く(15条項×3タイル、timeout宣言26〜56秒)、200が返った後も
スロットが数十秒塞がり続ける。対策は「リクエスト数×クエリコストの総量」を下げること。

対象は全て `js/legacy/part8.js`(条項リストはOSM_TILE_CLAUSES、part8.js:300付近)。
以下を優先順に実装してください。

## 1. 429時に /api/status を読んで待ち時間を正確化【最優先】

- fetchOSMTileBatchのcatch(429検知時)で `https://overpass-api.de/api/status` をGETし
  (レート制限の消費対象外・軽量)、応答テキストの `Slot available after: ...` 行から
  待ち秒数をパースして、グローバル・クールダウン(修正1のosmGlobalCooldownUntil)の
  待ち時間に使う。パース失敗時・行が無い時は現行の指数バックオフにフォールバック。
- `Rate limit:` と `N slots available now` もパースしてコンソールに1行ログ
  (デバッグ計器と統合可)。
- 注意: overpass-api.deはDNSラウンドロビンで、statusと直前のPOSTが別バックエンドに
  当たることがある(実測確認済み)。精度は完全ではない前提で、得られた待ち時間が
  現行の指数バックオフより短くても採用してよい(公式は「429なら15秒待って再送」を
  明記しており、現行の120秒上限はむしろ長すぎる可能性がある)。

## 2. クエリコスト削減【クールダウン自体を縮める唯一の手段】

- (a) `[timeout:]`宣言値を下げる: 1枚クエリ=15秒、3枚まとめ=25秒程度へ
  (buildOSMBatchQuery)。クライアントabort猶予は宣言値+8秒連動済みなので自動追従。
  504(宣言リソース過大)の受理率も上がる。
- (b) `[maxsize:]`を明示宣言: 未宣言だとデフォルト512MBで見積もられる。
  `[maxsize:33554432]`(32MB)を追加。1タイル応答の実サイズに対して十分か、
  密集地(東京駅・NY)のキャッシュ済み応答サイズを確認してから確定すること。
- (c) 条項の2段階分割: OSM_TILE_CLAUSESを
  「必須」(highway / building / relation building / railway / natural=water /
  waterway / relation water / 駅ノード)と
  「装飾」(landuse / leisure / amenity=school等 / natural=wood)に分け、
  必須クエリを従来のタイルパイプラインに流し、装飾クエリは別の低頻度キュー
  (例: 必須が取れたタイルに対して、キュー空き時のみ・現在地±1タイル優先で後追い)にする。
  processTileDataは同じ関数で両方を処理できる(seenOSMWaysで重複排除済み)はずだが、
  roadReadyTiles入りの判定は**必須クエリの成功時点**で行うこと。
  装飾未着のタイルを追跡する新Set(decorPendingTiles等)を作り、二重取得を防ぐ。

## 3. OSM_TILE_CONCURRENCY 3→2【5分の修正・単独での効果は期待しない】

- 3本目はスロットが空いていない限りほぼ常に429になる無駄弾で、failカウントと
  クールダウンを進めるだけ。悪化要因の除去として2へ。
- 実効スループットはスロット律速なので低下はほぼ無い見込み(むしろ無駄弾が消える分改善余地)。

## 4. 変更しないもの

- 実装済みのグローバル・クールダウン(修正1)、gaveUp再設計(修正5)、
  現在地ミラー緊急経路(修正7b、実装済みなら)は維持。役割分担は変わらない。
- バッチサイズの既定(3枚/近傍・失敗時1枚)は現状維持。2(c)の分割で1クエリが
  軽くなった後に、必要ならバッチ拡大を再検討する。

## 検証

1. デプロイ後、429発生時のコンソールで status由来の待ち秒数が出ること、
   待ち明けの再試行が成功しやすくなったこと。
2. 移動継続テストで、赤(fetching)の広がりが従来より狭いこと。
3. 2(c)実装時: 道路・建物・水は従来同等の速さで出て、公園面・木などの装飾が
   少し遅れて追いつく見た目になること(退行ではなく仕様)。
