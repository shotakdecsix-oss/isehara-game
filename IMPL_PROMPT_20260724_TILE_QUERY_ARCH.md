# 実装指示: タイルクエリアーキテクチャ改善(4フェーズ)

## 背景・目的

Overpassタイル取得の詰まり問題に対し、対症療法ではなく設計レベルの改善を4項目実施する。
設計検討の経緯は `ai-consult-tile-query-architecture-prompt.md` を参照(必要な結論は本書に含まれるので必読ではない)。

対象ファイル: クライアント=`index.html`(モノリシック)、サーバー=`server/server.js`。
デプロイ: git push で Render が自動デプロイ。手順・デプロイタイムスタンプの流儀は `DEPLOY.md` と既存実装に従う(タイムスタンプ更新を忘れない)。
ユーザーのPowerShellは5.x系のため、コマンド連結は `;` 区切りで提示する。

## 進め方(厳守)

**1フェーズずつ実装→commit/push→ユーザーの実機確認を待ってから次のフェーズへ。**
一括実装は禁止(リグレッション切り分けのため)。各フェーズ完了時にpushコマンドと確認観点を提示すること。

実装前に必ず現状コードを読み、本書の記述(関数名・仕組みの呼称)と実態が食い違う場合は実態を優先し、差分を報告すること。

## 前提知識(現状アーキテクチャ)

- タイル1600m四方。1タイル=道路・線路・建物・土地利用・水域を全部含む複合Overpassクエリ。
- 近傍(5x5)は1タイル単体クエリ、外周は最大3タイルまとめ。クライアント3並列。
- 優先度: 現在地ブロッキング > 近傍5x5 > それ以外、+待ち時間の年功序列。0.5秒ごとに再計算。
- サーバーはホスト単位の弾力レーン(バックログに応じ2〜4本)、ハードタイムアウト(平常45秒)、指数バックオフ、ディスクキャッシュ、同一クエリのinflight合流(重複排除)あり。
- クライアントの現在地タイルtimeoutは70秒(サーバーがinflightを束ねるため意図的に長い。短くしないこと)。
- 建物生成は該当タイルのroadReadyがゲート条件。未達の建物は隔離キューで待機。
- マップジャンプ時は進行中fetchのabort+キュークリア実施済み。
- 建物はqueue時点でbuildingGridに登録される(プロシージャル生成との競合防止。この不変条件を壊さないこと)。

## Phase 1: 近傍限定の道路/建物クエリ分離

**やること**: tier1〜2(現在地タイル+周囲1列=9枚)に限り、クエリを2段に分離する。

- 先行クエリ: 道路・線路・水域・土地利用(建物以外すべて)。軽量で1〜2秒で返る想定。
- 後追いクエリ: 建物のみ。先行クエリの完了(roadReady成立)後に発行。
- tier3以遠(周囲2列目〜)は現状の複合クエリ1本のまま変更しない。
- キャッシュキーは先行/建物で別にする。既存の複合クエリキャッシュとも衝突しないこと(クエリ文字列ハッシュがキーなら自然に分かれるはず。要確認)。
- 失敗カウント・リトライ・gaveUp判定も先行/建物で独立させる。建物クエリの失敗で道路を再取得しない。
- 移動によりタイルがtier2圏内に入った時、複合クエリで取得済み(キャッシュ有)ならそれを使い、分離クエリを重複発行しない。
- roadReadyゲート・隔離キュー・buildingGrid登録のタイミングとの整合を確認すること。

**受け入れ基準**: 密集地(例: 東京駅、香港)へのジャンプ直後、現在地周辺の道路が従来より明確に早く出る。建物は後追いで埋まる。デバッグタイルオーバーレイでgaveUp(紫)が増えないこと。

## Phase 2: エポック(世代)管理

**やること**: マップジャンプごとにインクリメントする世代番号を導入し、古い世代の結果を構造的に破棄する。

- クライアントにグローバルなepochカウンタ。ジャンプ時に+1。
- すべてのタイルリクエスト・キュー項目・レスポンスハンドラにepochを刻印。処理時に現epochと不一致なら黙って破棄(ワールドに反映しない)。
- 既存のabort+キュークリアは残す。epochはabort漏れ・レース(Promise絡みの取りこぼし)への保険という位置づけ。
- サーバー→Overpassのinflightはabortしない。完走させてディスクキャッシュに書き、クライアント側の待ちだけ切り離す(現状挙動を確認し、もしクライアントabortが上流fetchのabortに連動しているなら、キャッシュ書き込みは完走するよう分離する)。
- サーバーは全プレイヤー共有のため、epochをサーバー側キューの破棄判定に使う場合はプレイヤー(接続)単位で管理すること。複雑になるならサーバー側は触らずクライアント側のみで完結してよい。

**受け入れ基準**: 連続ジャンプ(A→B→C即座に)しても、前の場所のタイル結果が新しい場所に混入しない。通常移動に影響なし。

## Phase 3: tier1専用レーン(サーバー)

**やること**: server.jsの弾力レーン(2〜4本)のうち1本を「現在地ブロッキングタイル」優先レーンにする。

- クライアントはリクエストに優先度ヒント(例: ヘッダ `X-Tile-Priority: blocking|near|far`)を付与。
- 予約レーンのジョブ選択規則: blockingが待っていれば必ずそれを取る。blockingが無ければ単体タイルクエリのみ取ってよい(複数タイルまとめクエリは取らない=レーンがすぐ空く状態を維持)。
- 他のレーンは従来通り全ジョブを扱う。
- レーン数が2本に縮退している時も予約規則は維持(1本が予約、1本が汎用)。
- 既存の詰まり検出タイムアウト・隔離キュー・inflight合流との整合を確認。

**受け入れ基準**: 外周の重いまとめクエリが走っている最中にタイル境界をまたいでも、現在地タイルの取得が数秒以内に開始される(サーバーログで確認可能にする)。

## Phase 4: キャッシュTTL延長

**やること**: サーバーディスクキャッシュの有効期限を大幅延長する。

- まず現状のTTL実装(有無・値)を確認して報告。
- Overpassキャッシュ: 14日。標高キャッシュ: 30日。
- Renderのファイルシステムがエフェメラルならデプロイでキャッシュが消える前提を確認し、その場合TTL延長はローリスクなのでそのまま適用。永続ディスクの場合は容量上限(例: 500MB、LRU削除)も併せて実装。
- クライアントIndexedDB側のTTLは変更しない(キャッシュキーは都市間衝突バグ修正済みの現行方式を維持)。

**受け入れ基準**: 同一エリア再訪時(デプロイを挟まない場合)にOverpassへの実リクエストがほぼゼロになることをサーバーログで確認。

## 共通の注意(リグレッション監視ポイント)

- 道路タイル停滞・逆ドーナツ・gaveUp増加(デバッグタイルオーバーレイで確認)
- プロシージャル建物が実建物の場所を先取りする競合(buildingGrid queue時登録の不変条件)
- 建物と線路の重なり禁止、道路上への建物配置禁止
- GPU上限(bMaxキャップ、メッシュ数)まわりは触らない
- 現在地タイルclient timeout 70秒を短縮しない
