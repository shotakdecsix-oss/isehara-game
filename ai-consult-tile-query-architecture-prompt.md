# 相談したいこと: 地形/道路・線路/建物のストリーミング取得アーキテクチャ

## 背景

ブラウザで動くプロシージャル生成のオープンワールド探索ゲームを作っている。実世界の地理データ(OpenStreetMap由来)をプレイヤーの移動に合わせてリアルタイムに取得し、その場で3Dワールドとして生成・描画する仕組み。サーバーはプロキシ+キャッシュとしてのみ機能し、実データはOverpass API(OSMデータを地理範囲・タグで問い合わせられる公開API)から取得している。

現状の実装で「データ取得がまだ完了していないタイルが長時間そのままになる(詰まる)」問題に何度か直面し、対症療法的にサーバー側のキュー並列化・タイムアウト調整・優先度調整で凌いできた。今回は個別のバグ修正ではなく、**そもそものクエリ粒度・依存関係・優先順位づけの設計として、もっと良いアーキテクチャがあるのでは**という観点で意見が欲しい。

## データレイヤーの特性

world生成には性質の異なる3種類のデータレイヤーがある。

**地形(標高)**: プレイヤー中心の高解像度グリッド(近距離)と、広域の低解像度グリッド(遠景)の2段構え。取得元は別の標高API(Overpassとは無関係)。タイル単位のキューには乗らず、プレイヤー位置ベースで独立して定期更新される。今回相談したい問題の直接の対象ではないが、他レイヤーとの整合(地形が届く前に建物を置くと高さが合わない、等)は関係する。

**道路・線路**: Overpassクエリで、緯度経度の矩形範囲(タイル)を指定してhighway/railway系のwayを取得する。1タイルは1600m四方。道路・線路はプレイヤーの移動可能性・建物配置の可否(道路上に建物を置かない)に直結するため、他のどのレイヤーよりも早く確定してほしい。

**建物**: 同じくOverpassからbuilding系のway/relationを取得する。現在の実装では**道路・線路と同じクエリの中に混ぜて1回のリクエストで一緒に取ってきている**(理由: 元々は「タイル1枚=1回のAPI呼び出し」で完結させたかったため)。建物の生成は「そのタイルの道路データが確定していること」をゲート条件にしている(道路の直後に建物を置くと道路と重なるバグを防ぐため)。

## 現状のアーキテクチャ(要点)

- タイル単位(1600m四方)でOverpassにクエリを投げる。1回のクエリに道路・線路・建物・土地利用・水域など全種別のタグ条件をまとめて含める。
- プレイヤー近傍(現在地から一定距離以内)は1タイル単体でクエリし、外周は最大3タイルをbboxごとにまとめて1クエリにする(まとめた方がリクエスト数は減るが、1回のクエリが重くなり応答が遅くなる/失敗しやすくなるトレードオフがある)。
- クライアントは最大3並列でリクエストを出す。優先度は「現在地を直接ブロックしているタイル」>「近傍(5x5タイル程度)」>「それ以外(先読み範囲)」の3階層+同階層内では待ち時間による年功序列(飢餓防止)。同心円状の同心方眼で言えば、tier1=現在地タイル(1枚)、tier2=周囲1列目(8枚)、tier3=周囲2列目(16枚、tier1〜3合計で5x5=25枚)、tier4=それより外側、という4段階の明示的な優先度階層として捉えることもできる(現状は「近傍5x5」をひとまとめの階層として扱っており、tier2とtier3を区別していない)。
- サーバーはOverpassへの実際のリクエストをプロキシしており、ホスト単位で並列レーン(バックログに応じて2〜4本に弾力的に増減)を持つ。1回のリクエストにはハードタイムアウトがあり(平常時45秒、キューが詰まっている時は短縮)、失敗したら指数バックオフで再試行する。同一クエリのキャッシュ(サーバー側ディスク+クライアント側IndexedDB)もある。
- 建物生成は、道路データが届いたタイル(roadReady)についてのみ開始される。届いていないタイルの建物は「隔離キュー」で待機する。

## 制約条件

- Overpass公開インスタンスは1 IPあたり同時接続2本程度が経験的な上限で、それを超えると429(レート制限)や504が増える。
- クエリが重い(タイル数が多い・タグ条件が多い)ほど、Overpass側の処理時間が伸びる。実測では1タイル単体は1〜2秒で返るが、3タイルまとめは密集地(高密度な市街地)だと10〜30秒かかることがあり、まれに1リクエストが十数分かかることもある(公開APIの混雑・サーバー側の内部キューイングに起因すると思われる)。
- モバイル端末を含む低スペック環境が前提で、メッシュ数(ドローコール数)にGPU負荷の上限があるため、無制限にデータを先読み・保持できない。
- プレイヤーは「地続きの移動」だけでなく「瞬間移動(マップジャンプ)」もでき、その際は必要なタイル範囲が一瞬で全く別の場所に切り替わる。ジャンプ前の古いリクエストがまだ処理中に残っていると、新しい場所のリクエストがその後ろに並んで詰まって見える、という問題が過去にあった。
- 通常の地続き移動でタイル境界をまたぐ場合(マップジャンプほど極端ではない): 一定周期(0.5秒ごと)でプレイヤーの現在地を再チェックし、まだキューに無い新規タイルを追加している。キュー自体はクリアしない(前のタイルへのリクエストはそのまま進行を続け、優先度が「近傍」から「それ以外」に下がるだけ)。優先度スコアはプレイヤーの現在地を基準に毎回動的に再計算するため、tier境界(現在地タイル・周囲1列目・周囲2列目、という区分そのもの)もプレイヤーの移動につれて相対的に動く。ただし、既にサーバー側のレーンに入って処理が始まっている(inflightな)リクエストの優先度・実行順は変更できないため、タイルをまたいだ直後は「新しく最優先になったはずのタイル」が「もう優先度が下がったが処理が進行中の前のタイル」を追い越せない、という制約がある。また、タイル境界のすぐ近くを往復するような細かい移動をすると、tierの割り当てが頻繁に切り替わり、優先度スコアが小刻みに揺れる可能性がある(実害として顕在化しているかは未検証)。
- サーバーは全プレイヤー共有(1プロセス)で、キュー・キャッシュもグローバル。
- **同時接続数のスケーラビリティが未検証**: 現状は「ホスト単位で最大2〜4本のレーン」を全プレイヤーで共有する設計なので、専用の経路がプレイヤーごとに増えるわけではない。同時接続が多人数(例: 100人規模)になった場合、プレイヤー同士が同じような場所に密集していれば、サーバー側ディスクキャッシュと「同一クエリへの同時リクエストを1本に合流させる」重複排除の効果でユニークなタイル数に収束し、そこまで悪化しない可能性がある。一方、各プレイヤーがバラバラの場所を探索している場合は、ユニークなタイルへのリクエストが人数分積み上がり、最大2〜4本のレーンを奪い合う形になって待ち時間が大きく伸びる。さらにOverpass公開インスタンス自体に「1IPあたり同時接続2本程度」という上流側の制限があるため、サーバー側のレーン数をいくら増やしてもいずれこの壁に当たり、青天井にはスケールしない。つまり無料の共有Overpass APIに依存する構造そのものが、多人数同時プレイに対するスケーラビリティの天井になっている。

## 今のところ検討していて未着手の案

道路・線路クエリと建物クエリを完全に分離し、まず道路・線路だけの軽量なクエリを先に投げ、それが確定してから建物クエリを別途投げる、という2段階構成にする案。

想定されるメリット: 道路データだけなら建物込みより軽くなるはずで、道路の確定(プレイヤーの移動可否判定に直結)が早まる。建物クエリだけが失敗しても道路の再取得は不要になる。

想定されるデメリット: リクエスト数が単純に倍増し、Overpassのレート制限に当たりやすくなる可能性がある。タイルの状態管理・キャッシュキー・失敗カウント・リトライを道路用/建物用で別ライフサイクルに分ける実装コストがかかる。

もう一つ検討中なのが、優先順位(どのタイルを先に処理するか)とクエリ粒度(何タイルまとめて1回のリクエストにするか)を分けて設計する案。上記のtier1〜tier4を使うなら、例えば「tier1+tier2(9枚)は1タイル単体クエリで最速に、tier3(16枚)は優先度は高いまま維持しつつ2〜3タイルまとめてクエリしてリクエスト数を抑え、tier4は今まで通り大きくまとめる」といった組み合わせが考えられる。現状は「近傍5x5は全部1タイル単体クエリ」という単純な二値なので、tier3の16枚がまるごと個別リクエストになり、リクエスト数が増えている懸念がある。

## 相談したいこと

1. 「地形→道路・線路→建物」のように依存関係のあるレイヤーを、外部APIの矩形クエリでストリーミング取得する設計として、一般的にどういうパターン・ベストプラクティスがあるか。
2. 上記の「道路/建物クエリ分離」案は妥当か。他にもっと良い粒度の切り方(例: タイルサイズ自体を可変にする、レイヤーごとにタイルサイズを変える、優先度をレイヤー単位でも持つ、など)はあるか。
3. 「1件のリクエストがまれに極端に遅くなる(裾の長い分布)」外部APIに対して、キュー・タイムアウト・リトライをどう設計するのが定石か(サーキットブレーカー、適応的タイムアウト、投機的リトライ等、既存のパターンで参考になるものがあれば知りたい)。
4. マップジャンプのような「関心範囲が瞬間的に切り替わる」ケースで、進行中のリクエスト・キューをどう扱うのが良いか(今は进行中fetchのabort+キュークリアで対応しているが、もっと良い設計があるか)。
5. 上記のtier1〜tier4のような同心円状の優先度階層の切り方は妥当か。階層の数・境界(1枚/8枚/16枚/それ以遠、というChebyshev距離ベースの区切り)は適切か、他に良い切り方があるか。また、優先順位(処理順)とクエリ粒度(単体クエリかまとめクエリか)を階層ごとに別々に決める、という考え方自体は妥当か。
6. 通常の地続き移動でタイル境界をまたぐたびに優先度(tier)が動的に再計算される設計について、「既にinflightのリクエストは優先度が下がっても追い越されない」「境界付近を往復すると優先度が細かく揺れる」という点は許容して良い設計上の妥協か、それとも対処すべきか。対処するとしたら一般的にどういう手法があるか(ヒステリシス、tier境界の遅延切替、inflightリクエストのキャンセルによる再優先順位付け、等)。
7. 同時接続プレイヤー数が増えた場合(例: 100人規模)のスケーラビリティについて、無料の共有外部API(1IPあたり同時接続2本程度という上流側の制限がある)に依存したこのアーキテクチャは、根本的にどこで頭打ちになるか。キャッシュ・重複排除だけでは吸収しきれない負荷増(プレイヤーが分散して探索するケース)に対して、サーバー側でどう緩和・優先順位付け・劣化のさせ方(グレースフルデグラデーション)を設計するのが定石か。

実装コードそのものではなく、設計・アーキテクチャの観点でのアドバイス・トレードオフの整理が欲しい。
